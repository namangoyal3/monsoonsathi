# MonsoonSathi — §26 Final Response (Submission Report)

> Status: verification and production sections contain only results from executed runs.

## Implemented

Only features that genuinely work end-to-end against live services:

- **Monsoon action planner form** (`app/page.tsx`): locality, scope (individual / family / community), phase (before / during / after), language (English / Hindi / Kannada), optional travel destination. Demo chips pre-fill fields only — every successful plan runs the production pipeline and requires live Gemini output.
- **Live location + weather** (`lib/geocode.ts`, `lib/weather.ts`): OpenWeather geocoding, current conditions, and forecast; a four-minute public-data cache (`lib/cache.ts`) limits provider traffic while preserving the provider observation timestamp.
- **Live Gemini structured output** (`lib/gemini.ts`, `lib/prompt.ts`): configurable server-side model (default `gemini-flash-lite-latest`), `responseMimeType: application/json`, compact `responseJsonSchema`, temperature 0.3. Invalid output gets one real Gemini repair attempt, never deterministic plan text.
- **Deterministic validation wall** (`lib/schema.ts`, `lib/validate-plan.ts`): Zod parse, enum coercion only, completeness assertions (support actions when family/community flags set; travel section when destination given), rejection of invented phone numbers, unknown source IDs, model HTML/URLs, unsupported route-safety claims, and travel `go` recommendations.
- **Evidence grounding** (`lib/ndma-guidance.ts`): versioned NDMA guidance bullets passed to the model as citable source IDs; every factual action must cite supplied evidence of the matching kind.
- **Labeled dashboard**: every card distinguishes *Live weather fact* vs *AI-generated guidance*; travel card carries a deterministic disclaimer.
- **Security envelope** (`lib/rateLimit.ts`, `next.config.ts`, `lib/env.ts`): 30 plan POSTs/min/IP per instance, 32 KB streamed-body cap, 55-second shared request deadline, CSP + nosniff + frame-deny + HSTS headers, server-only API keys.
- **Honest failure states**: invalid/incomplete model output → structured error response; no canned plan fallback anywhere in the codebase.

## Live GenAI use

Gemini has exactly **one production integration path**: `POST /api/plan` → `lib/gemini.ts`.

Every successful plan is generated by at least one live structured-output call, which produces: interpretation, prioritization rationale, do-now/do-next actions, adaptive checklist, selected-phase actions, other-phase summaries, family/community support actions, travel guidance (without flood-safe claims), assumptions/limitations — in the requested language. A failed safety/schema check may trigger one additional live repair attempt, and a transient upstream failure (quota burst / 5xx / network blip) gets one bounded real retry — every retry is a live Gemini call, never a canned response. `modelCalls >= 1` is required for `ok: true`; there is no path that returns a plan without live Gemini output. Deterministic code never authors guidance; it only validates, grounds, secures, caches, and displays it. See `docs/GENAI_USAGE.md`.

## Live data services

- **OpenWeather Geocoding API** — resolves the free-text locality (and optional travel destination) to coordinates on every request.
- **OpenWeather Current + Forecast APIs** — live conditions and short forecast injected into the evidence packet and rendered as labeled facts.
- **OpenWeather One Call alerts** — best-effort publisher-alert lookup; absence or plan-level unavailability is labeled honestly.
- **OSRM public routing** — traffic-unaware car time/distance estimate. Other modes use straight-line context; neither path asserts road or flood safety.

## Security

Summarized from `docs/SECURITY.md`:

- Untrusted profile text is delimited and the system prompt forbids following instructions inside it (prompt-injection guard, covered by tests).
- Semantic validators reject invented emergency numbers, out-of-set source IDs, evidence-kind mismatches, HTML/URLs, flood-safe-route phrasing, and travel `go` recommendations.
- Keys are server-only (`GEMINI_API_KEY`, `OPENWEATHER_API_KEY`); no `NEXT_PUBLIC_*` secrets.
- Rate limit 429 + `Retry-After`; request size cap; strict security headers; React text-only rendering (no `dangerouslySetInnerHTML`).
- Logs contain request IDs, error codes, timings only — no profiles, prompts, coordinates dumps, or keys.

## Known limitations

- Rate limiting is per-instance memory, not a distributed limiter.
- Weather cache TTL means conditions can lag by a few minutes.
- Hindi/Kannada output quality depends on the model; validators check structure and safety, not fluency.
- No offline mode, user accounts, or server-side persistence. Opt-in device memory stores only allowlisted basic preferences; it excludes destination, support needs, personal notes, and plans.
- Community scope generates guidance for a generic community profile; it does not ingest real community rosters.
