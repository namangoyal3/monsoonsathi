# MonsoonSathi — §26 Final Response (Submission Report)

> Status: verification and production sections contain only results from executed runs.

## Implemented

Only features that genuinely work end-to-end against live services:

- **Monsoon action planner form** (`app/page.tsx`): locality, scope (individual / family / community), phase (before / during / after), language (English / Hindi / Kannada), optional travel destination. Demo chips pre-fill fields only — every submit hits live APIs.
- **Live location + weather** (`lib/geocode.ts`, `lib/weather.ts`): OpenWeather geocoding and current weather on every plan request; short TTL cache (`lib/cache.ts`) to stay inside free-tier limits without serving stale-day data.
- **One Gemini structured-output call per plan** (`lib/gemini.ts`, `lib/prompt.ts`): `gemini-flash-lite-latest`, `responseMimeType: application/json`, compact `responseJsonSchema`, temperature 0.35.
- **Deterministic validation wall** (`lib/schema.ts`, `lib/validate-plan.ts`): Zod parse, enum coercion only, completeness assertions (support actions when family/community flags set; travel section when destination given), rejection of invented phone numbers, unknown source IDs, model HTML/URLs, and "flood-safe route" claims.
- **Evidence grounding** (`lib/ndma-guidance.ts`): versioned NDMA guidance bullets passed to the model as citable source IDs; plan citations must be a subset.
- **Labeled dashboard**: every card distinguishes *Live weather fact* vs *AI-generated guidance*; travel card carries a deterministic disclaimer.
- **Security envelope** (`lib/rateLimit.ts`, `next.config.ts`, `lib/env.ts`): 8 plan POSTs/min/IP, payload size cap, CSP + nosniff + frame-deny + HSTS headers, server-only API keys.
- **Honest failure states**: invalid/incomplete model output → structured error response; no canned plan fallback anywhere in the codebase.

## Live GenAI use

Gemini is called in exactly **one place**: `POST /api/plan` → `lib/gemini.ts`.

Every successful plan is generated by that single live structured-output call, which produces: interpretation, prioritization rationale, do-now/do-next actions, adaptive checklist, selected-phase actions, other-phase summaries, family/community support actions, travel guidance (without flood-safe claims), assumptions/limitations — in the requested language. `modelCalls >= 1` is required for `ok: true`; there is no path that returns a plan without a live Gemini call. Deterministic code never authors guidance; it only validates, grounds, secures, caches, and displays it. See `docs/GENAI_USAGE.md`.

## Live data services

- **OpenWeather Geocoding API** — resolves the free-text locality (and optional travel destination) to coordinates on every request.
- **OpenWeather Current Weather API** — live conditions injected into the evidence packet and rendered as labeled facts.
- No Routes/Directions API is used; the app deliberately refuses road-safety claims instead of pretending to know them.

## Security

Summarized from `docs/SECURITY.md`:

- Untrusted profile text is delimited and the system prompt forbids following instructions inside it (prompt-injection guard, covered by tests).
- Semantic validators reject invented emergency numbers, out-of-set source IDs, HTML/URLs, and flood-safe-route phrasing.
- Keys are server-only (`GEMINI_API_KEY`, `OPENWEATHER_API_KEY`); no `NEXT_PUBLIC_*` secrets.
- Rate limit 429 + `Retry-After`; request size cap; strict security headers; React text-only rendering (no `dangerouslySetInnerHTML`).
- Logs contain request IDs, error codes, timings only — no profiles, prompts, coordinates dumps, or keys.

## Known limitations

- Rate limiting is per-instance memory, not a distributed limiter.
- Weather cache TTL means conditions can lag by a few minutes.
- Hindi/Kannada output quality depends on the model; validators check structure and safety, not fluency.
- No offline mode, no user accounts, no persistence — by design for the competition MVP.
- Community scope generates guidance for a generic community profile; it does not ingest real community rosters.
